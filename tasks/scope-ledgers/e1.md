# E1 Scope Ledger

## authorized

- Implement S1 steps 1-16 from `tasks/spec.md` as one shared scenario using
  the E0 runner and business-action boundaries.
- Cover the main Root journey, Plan approval barrier, ordered Work/Human
  handling, Root Gate and delivery, UI evidence, and two API Key Profiles with
  one bounded secret frame per Profile.
- Add the E1 scenario command and dry-run acceptance entrypoint required by
  `tasks/plan.md`.

## required_consequences

- No Work Turn or commit occurs before Plan Approval is Done.
- Human Work does not advance before the operator answer is read back.
- At most one Performer Turn is active; Root Gate succeeds before delivery.
- Root A reaches a terminal V1 delivery state before Root B becomes runnable.
- Root A keeps its primary Profile; Root B fixes the active secondary Profile
  and uses its updated model/reasoning settings with Fast disabled.
- Secrets and repository paths do not enter evidence, views, logs, or custom
  persistence; Codex-owned configuration remains outside Symphony access.

## out_of_scope

- S2 authority probes, S3 recovery/fault injection, Linux CI, Windows UI,
  ChatGPT login, V2 multi-Root scheduling, a second Provider, nightly runs,
  automatic merge, or compatibility paths.
- New product workflow state, persistence, public contracts, or customer-facing
  authentication behavior.

## assumptions_requiring_approval

- None. The user authorized implementing all E2E tasks before supplying the
  deterministic live fixture and attempting the real external run. Missing
  fixture inputs must fail closed and must not produce a passed verdict.

## deferred_ideas

- S2/S3 probes, CI orchestration, ChatGPT evidence, Windows support, and
  dependency toolchain migration remain assigned to later or excluded work.
- The concrete Root A/Root B inputs, repository starting state, expected
  Provider results, and successful real S1 run are deferred until all E2E tasks
  are implemented, as explicitly authorized by the user.
