# E1 Review

## Scope

Reviewed the E1 diff against S1 steps 1-16 in `tasks/spec.md`, the E1 scope
ledger, and the Roadmap, Linear flow, Root Issue, Performer Profile, and Podium
Desktop architecture sources.

## Code simplification

- Accepted: removed the coordinator's accumulated raw observation context.
  Later steps did not consume it, and retaining it bypassed the sanitized
  evidence boundary.
- Accepted: Profile lookup takes an explicit display name instead of mutating
  one implicit current-Profile variable. This directly supports two Profiles
  with less hidden state.
- Retained: the fixed step table and named predicates. They make every S1
  barrier inspectable and keep orchestration separate from platform UI actions.
- No unrelated product or architecture refactors were accepted.

## Five-axis review

### Required and accepted

- Correctness: Plan stable-window failure stops before approval, and Root Gate
  failure stops before delivery. Both have negative-control tests.
- Correctness: Root B evidence requires Root A terminal, unchanged Root A
  Profile, secondary Root B Profile, updated settings, and Fast disabled.
- Security: removed raw observations from the returned scenario result.
- Security/correctness: contract execution reports `completed`, never `passed`;
  only future real-boundary evidence may produce a passed verdict.
- Readability: added a closed reasoning-effort-to-visible-label mapping at the
  WDIO UI boundary.
- Correctness: secondary Profile creation uses the same bounded secret frame,
  activates independently, then applies fixture-provided settings.
- Safety: dry-run performs no mutation, and live invocation remains explicitly
  opt-in and fail-closed while the fixture/driver is unavailable.

### Deferred by explicit user authorization

- Concrete Root A/B content, repository starting state, expected Provider
  results, and the real S1 Linear/Codex/Desktop run. The user authorized
  implementing all E2E tasks before supplying fixtures and attempting live
  execution. Their absence must keep the verdict incomplete.

### Rejected as scope expansion

- S2 authority probes, S3 recovery faults, CI workflow changes, Windows UI,
  ChatGPT live login, V2 multi-Root scheduling, a second Provider, automatic
  merge, compatibility paths, and product contract/persistence changes.
- Inventing a model name or Provider fixture output. Secondary settings remain
  fixture inputs instead of becoming new product defaults.

## Residual risk

The S1 orchestration and Desktop Profile controls are contract-tested, not
validated against real Linear, Codex, GitHub, or packaged Desktop boundaries.
`acceptance:v1 -- --scenario S1` therefore exits blocked and cannot produce a
passed verdict until the deferred live fixture and driver are configured.
