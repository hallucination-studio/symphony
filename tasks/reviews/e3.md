# E3 Review

## Scope

Reviewed E3 against `tasks/plan.md`, `tasks/spec.md`, the E3 scope ledger, the
Roadmap acceptance boundary, and repository CI/CD rules.

## Code simplification

- Accepted: `acceptance:v1 -- --preflight` delegates to the canonical doctor
  instead of introducing a second configuration validator.
- Accepted: the CLI uses an early-returning `main()` so preflight and scenario
  modes remain visibly separate.
- Accepted: one small split-verdict formatter wraps the existing evidence
  evaluator without duplicating artifact validation.
- Retained explicit S1, S2, S3 workflow steps because their serial order is an
  acceptance invariant, not generic matrix work.
- Rejected a general cleanup framework; E3 only owns the run lock currently
  created by the E2E runner.

## Five-axis review

### Required and accepted

- Correctness: workflow is manual-only, uses `ubuntu-24.04`, binds the
  `roadmap-v1-e2e` Environment, and serializes HELL with one non-canceling
  concurrency group.
- Correctness: CI clones the allowlisted dedicated E2E repository, runs shared
  preflight/build entrypoints, then invokes S1, S2, and S3 in fixed order under
  Xvfb.
- Security: GitHub secrets are step-scoped rather than job-scoped. Build,
  evidence evaluation, artifact upload, and cleanup do not receive operator or
  Provider credentials.
- Security: `@wdio/tauri-service` merges `process.env`; therefore WDIO now
  explicitly removes user Linear, OpenAI, GitHub, and `GH_TOKEN` values from the
  spawned app process. A negative control guards this boundary.
- Security/correctness: cleanup validates the run ID and removes a lock only
  when its embedded owner exactly matches the current run. Another run's lock
  is preserved.
- Correctness: evidence collection, evaluation, upload, and cleanup use
  `always()`; collection success remains distinct from acceptance success.
- Correctness: automated API Key evidence and full Roadmap status are split.
  Even fully passing API Key rows leave Roadmap V1 incomplete with
  `chatgpt_live_login_not_run`.
- Readability: workflow contract tests verify trigger, Environment, platform,
  concurrency, command order, secret mappings, and always-run handling.

### Deferred by explicit user authorization

- GitHub Environment creation and reviewer assignment, concrete S1-S3 live
  fixtures, successful Linux workflow execution, and real artifact inspection.
  Current live commands remain fail-closed and cannot produce passed evidence.

### Rejected as scope expansion

- Push, pull-request, schedule, nightly, or soak triggers; macOS/Windows CI;
  ChatGPT live login; deployment/rollback automation; automatic merge; V2
  scheduling; external-service chaos; and repository code that configures
  GitHub Environment reviewers or branch protection.
- Unrelated dependency upgrades for existing audit advisories with no available
  fix. They remain recorded from E0 rather than expanding E3 into a toolchain
  migration.

## Residual risk

The workflow is schema-parseable and contract-tested locally but has not run on
GitHub-hosted Linux. The Environment and secrets are external configuration.
Because live fixtures/drivers are intentionally deferred, the first reviewed
workflow run is expected to remain blocked until those inputs are configured.
Local production packaging completed through the macOS DMG artifact; Linux
packaging remains covered only by the unexecuted workflow.
